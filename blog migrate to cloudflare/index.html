# GitHub Pages? 该滚蛋了

好吧，我承认。我的个人网站（或者说博客，随便你怎么叫）在 GitHub Pages 上已经待了... 多久了？我不记得了，反正很久了。

这本身不是什么问题。GitHub Pages 能工作，它确实能工作。但问题是，它只是"能工作"而已。就像你用一把钝刀也能切菜，但为什么不换把锋利的？

## 为什么是 Cloudflare？

让我直接说：**自定义域名**。就这么简单。

GitHub Pages 当然支持自定义域名，但 Cloudflare 做得更好。而且，Cloudflare Pages 是免费的，性能更好，CDN 全球分布，部署速度更快。最重要的是，我已经在 Cloudflare 上管理其他域名了，为什么不把所有东西放在一个地方？

这不是什么技术革命，这只是**常识**。当你有了更好的工具，你就应该用更好的工具。这不是背叛，这是进化。

## AI 让我不再拖延

说实话，这个决定我拖了很久。迁移网站？听起来就像是要花一个周末的时间去折腾那些该死的 DNS 配置和部署脚本。

但这次不一样。我让 AI 帮我写迁移脚本，检查配置，甚至帮我写这篇博客（虽然我肯定会改得面目全非）。结果？整个过程只花了我几个小时，而不是一个周末。

**这就是 AI 最大的好处：它让你不再拖延。**

不是因为它能替代你思考，而是因为它能快速处理那些你明知道该做但一直拖着不做的琐事。DNS 配置？让 AI 生成，你检查一下就行。部署脚本？让 AI 写，你测试一下就行。

这不是偷懒，这是**效率**。

## 限期一个月

我给自己设了个期限：**一个月**。

一个月后，GitHub Pages 上的网站就关闭了。如果你还在用旧的链接，那是你的问题，不是我的。我已经给了你足够的时间去更新书签。

这个期限不是随便定的。一个月足够：
- 更新所有外部链接
- 通知那些还在用旧链接的人（如果他们真的在乎的话）
- 确保新站点稳定运行
- 处理任何迁移问题

如果一个月后你还在抱怨链接失效，那说明你根本不在乎这个网站，所以我也没必要在乎你的抱怨。

## 技术细节（给那些想知道的人）

迁移过程其实很简单：

1. **在 Cloudflare 上创建 Pages 项目**
   - 连接 GitHub 仓库
   - 设置构建命令（如果有的话）
   - 配置自定义域名

2. **更新 DNS 记录**
   - 添加 CNAME 或 A 记录指向 Cloudflare Pages
   - 等待 DNS 传播（通常几分钟到几小时）

3. **测试，测试，再测试**
   - 确保所有页面正常加载
   - 检查 HTTPS 证书（Cloudflare 自动处理）
   - 验证自定义域名工作正常

4. **关闭 GitHub Pages**
   - 一个月后执行
   - 删除 GitHub Pages 设置
   - 清理旧的部署

就这么简单。如果你觉得复杂，那是因为你把它想复杂了。

## 最后的话

这不是什么重大公告，这只是我个人的技术决策。如果你觉得有用，可以参考。如果你觉得我在浪费时间，那就别看。

但如果你也在用 GitHub Pages，而且已经用了很久，也许你也该考虑迁移了。不是因为 GitHub Pages 不好，而是因为**有更好的选择**。

而且，现在有 AI 帮你，你还有什么借口拖延？

我已经警告过你了。
